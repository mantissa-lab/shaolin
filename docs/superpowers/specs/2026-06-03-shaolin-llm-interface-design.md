# LLM / agent interface — Design Spec

**Date:** 2026-06-03
**Status:** Draft — pending review
**Parent:** [shaolin framework design](2026-06-03-shaolin-framework-design.md)
**Depends on:** [shaolin-core](2026-06-03-shaolin-core-design.md), [shaolin-cli](2026-06-03-shaolin-cli-design.md), [agent-ownership](2026-06-03-shaolin-agent-ownership-design.md)
**Sub-project:** 11 of 11 (the dedicated LLM-operability surfaces)

## 1. Purpose

LLM-friendliness is a cross-cutting principle (parent §4.6); this sub-project delivers the
**dedicated surfaces** that make shaolin operable end-to-end by an AI agent: a machine-readable
description of the whole app, an **MCP server** exposing shaolin's capabilities as tools, generated
agent-orientation docs, exported schemas, and a uniform actionable-error contract.

The goal: an agent can discover the system, make a scoped change to one module, verify it, and ship
it — entirely through structured tools and stable conventions, never by scraping shell output.

## 2. Foundation (verified 2026-06-03)

- **Official Ruby MCP SDK** — the `mcp` gem (v0.14.0, Apr 2026; maintained with Shopify):
  `MCP::Server` (JSON-RPC core), `MCP::Tool` (define tools), **Stdio and HTTP** transports;
  depends on `json-schema`.

> Exact `mcp` 0.14 API and Ruby 4.0 compat confirmed at planning.

## 3. `shaolin describe [--json]` — the whole-app map

Emits, from the module registry + manifests + introspection, a single structured document:

```jsonc
{
  "app": "...", "ruby": "4.0.5",
  "modules": [{
    "name": "users",
    "owner": "...",
    "commands":  [{ "name": "RegisterUser", "schema": { ... } }],
    "events":    { "published": [{ "type": "users.user_registered", "version": 1, "schema": {…} }],
                   "subscribed": ["billing.invoice_paid@1"] },
    "exports": ["user_service", "queries.find_user"],
    "imports": ["notifications.mailer"],
    "http_routes": [{ "verb": "POST", "path": "/users", "action": "create" }],
    "read_models": ["user_record"]
  }],
  "errors": [ /* any boot/lint/contract problems, with fixes */ ]
}
```

An agent loads this instead of reading dozens of files — the map of the entire system in one call.
A human-readable form is the default; `--json` is the agent form.

## 4. MCP server (`shaolin mcp`)

An `MCP::Server` exposing shaolin's capabilities as **tools** (Stdio for a local agent, HTTP for
remote):

| MCP tool | Wraps | Returns |
|---|---|---|
| `describe` | §3 | full app map (JSON) |
| `graph` | agent-ownership §7 | module dependency / blast-radius graph |
| `routes` | shaolin-cli | HTTP routes + Kafka subscriptions |
| `schemas` | §5 | command/event JSON schemas |
| `lint_isolation` | agent-ownership §4 | violations with `module:file:line` + fix |
| `contract_check` | agent-ownership §6 | CONTRACT/schema drift + result |
| `module_test` | agent-ownership §5 | run one module's specs in isolation |
| `generate` | shaolin-cli | scaffold module/aggregate/command/... |

Tools return **structured results with embedded remediation**, so an agent acts on outputs
directly. The server is read-mostly by default; mutating tools (`generate`) are explicitly flagged.

## 5. Exported schemas (`shaolin schemas [--json]`)

Every command and event has an explicit dry-schema/dry-struct definition; this command exports them
as **JSON Schema** so an LLM knows exact payload shapes before generating a command or producing an
event. Integration-event envelopes (shaolin-messaging) are included with their versions.

## 6. Agent-orientation docs (generated by `shaolin new`)

- `AGENTS.md` (+ a `CLAUDE.md` pointer) at repo root: what shaolin is, the deterministic
  conventions, the module workflow (`shaolin g module …`), how to run/verify a single module, where
  things live, and how to read a `CONTRACT.md`. Regeneratable; drift-checked.
- Per-module `CONTRACT.md` (from agent-ownership §3) is the local orientation.

## 7. Uniform actionable-error contract

A framework-wide error shape — `{ code, message, fix, doc }` — used by boot, lint, contract, HTTP,
and CLI errors. Every error tells an agent (or human) **what to do next**, not just what failed.
Sub-projects already specify fail-fast, named errors; this standardizes the *shape* so it is
uniformly machine-parseable.

## 8. Public surface

- CLI: `shaolin describe [--json]`, `shaolin schemas [--json]`, `shaolin mcp [--stdio|--http]`.
- `Shaolin::Describe` — builds the app map from the registry (pure, testable).
- `Shaolin::MCP::Server` — the MCP server wiring shaolin tools.
- `Shaolin::Error` — the uniform error contract (`code`/`message`/`fix`/`doc`).

## 9. Error handling

- `describe`/`schemas` on a non-booting app still emit partial output with the boot errors listed
  (an agent needs the map *especially* when something is broken).
- MCP tool failures return structured MCP errors carrying the same `{code, message, fix}` shape.

## 10. Testing strategy

- `describe --json` validated against its own JSON Schema across fixture apps.
- MCP tool contract tests: each tool invoked via the server returns the documented structure
  (using the `mcp` gem's test harness / a stdio round-trip).
- `AGENTS.md` generation + drift check tested.
- Error-contract shape asserted across a sample of every error source.
- RSpec; TDD.
